# Windows PE named-anchor research

## Status and evidence boundary

This is a targeted, read-only map for the locally inventoried Windows
executable. It records exact string locations and conservative address-literal
candidates. It does **not** recover proprietary source, prove function
boundaries, identify a PRNG, or transfer offsets to another build.

The durable machine-readable result is
`data/observatory/native/windows_build_13725832_31fe35265598_pe_anchors.json`.
It was generated by `scripts/itb_pe_anchor_map.py`, whose parser and mapping
logic are covered by synthetic PE tests.

## Direct PE facts and conservative candidates

All five requested ASCII strings occur once, null-terminated, in `.rdata`.
The mapper then searches executable sections for the pointer-sized virtual
address of each occurrence. It classifies common x86 operand byte layouts but
does not claim that every byte match is a decoded instruction or registration
site.

## Community evidence

The community work supports a narrow empirical strategy:

- The Mod Loader README at pinned commit
  [`cba102c7`](https://github.com/itb-community/ITB-ModLoader/blob/cba102c7f41ffec3ca55cc4b7fa964dab51d3a44/README.md)
  describes a Windows-only loader with extension DLLs and attributes its Lua
  proxy DLLs to AUTOMATIC1111. This demonstrates an existing Windows boundary,
  not transparent behavior for our exact build.
- The attributed historical proxy at pinned commit
  [`de060823`](https://github.com/AUTOMATIC1111/IntoTheBreachLua/blob/de0608234f8920379e968ba6efb41e85b0704b50/lua-hooks.cc)
  contains forwarding hooks for Lua API calls including `lua_pushcclosure` and
  `lua_setfield`. It is a feasibility reference, not proof that current Mod
  Loader binaries are source-equivalent or safe for tracing.
- Community memedit at pinned commit
  [`8f809a0`](https://github.com/itb-community/memedit/blob/8f809a0d69d4652e041a545f58c33a8927472d39/memedit.lua)
  stores address calibration by `modApi.gameVersion` and can recalibrate using
  controlled scanners. Its
  [`Board:GetAttackOrder`](https://github.com/itb-community/memedit/blob/8f809a0d69d4652e041a545f58c33a8927472d39/functions/board.lua)
  support is a useful independent phase/orchestration probe. Observatory work
  should strengthen the key from a version string to the full executable and
  content identity.
- KnightMiner's historical
  [public-function inventory](https://gist.github.com/KnightMiner/2f308a3747461748d6a2186d823e3424)
  is explicitly a mostly complete binary-derived list and predates full
  Advanced Edition coverage. It does not contain these five requested names,
  so absence is not evidence that the functions are unavailable. Adjacent Pawn
  names such as `GetCustomPositionScore`, `GetDangerScore`, and `GetTarget` are
  useful lead-generation anchors only.

The inventoried local `lua5.1.dll` is 419,840 bytes with SHA-256
`0157f0c34e72b32e63ebf3fdd9a21215de674b51b6d1750ebe545ef3093a0c14`.
At the pinned Mod Loader revision, that byte identity corresponds to its
preserved original rather than its smaller proxy DLL. Therefore no current
public-proxy hook should be assumed active in the local install, and nothing
should overwrite it during achievement play.

## Inferences, hypotheses, and next proof

### Inferences

- Following `random_int` and `random_bool` candidates to code that uses imported
  `lua_pushcclosure` plus `lua_setfield` is the shortest route to candidate
  `lua_CFunction` pointers.
- The two `ScorePositioning` candidates may represent separate orchestration
  paths. The single `GetTargetScore` candidate may be a callback lookup path.
  These names remain provisional until control flow confirms them.
- The two `aiSeed` candidates appear to be key/string handling sites. A key
  reference is not a live RNG-state address.

### Unresolved hypotheses

- `random_int`, `random_bool`, and `aiSeed` share state.
- Target tie-breaking consumes either random binding.
- Current Mod Loader proxy binaries preserve the historical proxy behavior.
- Community offsets for game version 1.2.93 apply to this executable merely
  because a version string matches.

### Safe next step

1. Create a Ghidra project named by the full identity above and enter the five
   string RVAs.
2. Confirm every candidate as code or data, recover containing function starts,
   and record imported Lua API calls without committing decompiled source.
3. For the random bindings, separate thin Lua-stack wrappers from downstream
   state/PRNG routines.
4. Map targeting candidates together with `GetCustomPositionScore`,
   `GetDangerScore`, `GetTarget`, `lua_getfield`, and `lua_pcall`.
5. Validate any boundary tracer only in a disposable clean copy or VM, pin all
   proxy hashes, and compare identical-input enabled/disabled trials before any
   bridge or live-install change.
