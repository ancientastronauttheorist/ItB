Blitzkrieg - Lightning War

Proof-Gated Sprint Plan

The achievement run is no longer treated as a live experiment. The controller only spends timer through named, prevalidated transactions with proof before and after. Unknown state means pause, stop, or reset.

Screenshot timer is ground truth LLM never thinks unpaused Esc is the only pause toggle Archive - Detritus - Pinnacle Wrong window means hard stop
< 30:00 Finish the first two islands by visible in-game timer.
~2:00 Target mission segment, including route and UI tail.
< 3:00 Hard ordinary mission ceiling. Over means reset.
< 0:20 Opening target: mechs placed and Confirm clicked.

Core Change

The plan is not simply to go faster. The plan is to go fast only through states that are proven safe, current, and game-focused. The runner should never treat a plausible route, stale screenshot, vague pause menu, or unknown UI as permission to continue.

1. Prove Focus Frontmost app, bounds, and screenshot identity must be Into the Breach.
2. Prove Pause Capture a fresh PausedFrame with timer and underlay class.
3. Execute Burst Esc resume, perform exact deterministic controls, obey max wall duration.
4. Snap + Esc Take screenshot, immediately pause, then classify the result.
5. Accept or Reset Continue only with post-burst proof; unknown state fails closed.

Hard Invariants

  • No game proof, no click. Codex, Desktop, Steam, macOS prompts, or stale frames cannot authorize game controls.
  • No generic pause authority. The controller must know what screen is under the pause menu before choosing the next action.
  • No stale execution. Combat plans require fresh bridge state; UI decisions require fresh screenshot proof.
  • No live deliberation. Codex never thinks unpaused; Python may only run deterministic live bursts with fixed preconditions.
  • No route optimism. Start Mission requires current region/preview proof and post-Start deployment proof; mission identity helps, but it is not a speed veto.
  • No hidden terminal panels. KIA, failed objective, grid death, timeline loss, region secured, and achievement pop must be classified before clearing panels.

Failure Means Stop

  • Wrong-window screenshot or mismatched game bounds.
  • Missing visible timer when timer should be visible.
  • Pause proof without a screen-family classification.
  • Stale preview, wrong-region proof, impossible candidate count, or no post-Start deployment signal.
  • Any ordinary mission segment over 3:00.
  • Bridge desync that one fresh read and re-solve cannot recover.
  • Unexpected reward/shop/map state that would require live reading.

Proof Objects

GameFocusProof

  • Frontmost app is Into the Breach.
  • Window title and bounds match calibration.
  • Screenshot contains game HUD or known game UI anchors.
  • Retina scale and window-local coordinate transform are current.

PausedFrame

  • Fresh screenshot path and capture timestamp.
  • Visible timer OCR, or strong pause-menu proof when timer is hidden.
  • Pause overlay proof plus stable game focus proof.
  • Underlay class: map, preview, deployment, combat, reward, shop, setup, or unknown.

BridgeFreshnessProof

  • Heartbeat age within the live threshold after unpause.
  • Mission id, phase, turn, active mechs, and grid snapshot.
  • Unit-roster fingerprint stable after enemy phase settles.
  • Explicit flag when a paused cached bridge read is being used by a Lightning wrapper.

Opening Race Ladder

Treat the first minute as a qualifying lap. A run that cannot reach deployment quickly is valuable data, but it is not an achievement attempt. Use Esc to reach the pause menu, return through Main Menu to setup, carry the evidence forward, and spend the next timer seconds on a cleaner UI opening.

0:03 Ideal First island click begins. Keep pushing toward this with fixed click bursts.
0:11 Proven Current patched runs can pause on Archive around this timer. Slower than 0:15 is a reset signal.
0:20 Target Mission committed and deployment/Confirm in progress. This remains the benchmark.
0:35 Reset No current preview-board commit or deployment handoff by here means Esc pause menu to Main Menu/setup, not one more preview.

Human Calibration

  • Speed source: the solver can tackle the mission slate; the timer savings come from knowing the UI state and firing the next transaction immediately.
  • Timer hypothesis: assume map, preview, reward, and shop screens spend timer until screenshots prove otherwise.
  • Route commit: click the highlighted 8x8 mission preview board when route proof is current. Keep Start-text clicks as experiments or fallback only.
  • Deployment: use the fast deployment helper/script as the deployment authority, then Confirm as soon as placement proof exists.
  • Mission policy: do not browse for easier missions. The solver can tackle the slate; unproven UI is the real blocker.
  • Shop policy: read Grid Power, spend only to max Grid Power, then leave immediately. Skip every other reward/shop optimization.
  • Success proof: offline ITB log, on-screen achievement pop, or the colored middle achievement tile in the pause menu can prove Lightning War.

First-Island Slate

  • Speed cycle: Archive, then Detritus, then Pinnacle. Repeat only after those slates fail.
  • Sequence: verify setup, click setup modal Start, fast-click first island, record pending island, clear intro, pause, read visible timer.
  • RST is data-only until its opening UI path has the same measured preview-board speed as the other slates.
  • Archive-only rerolls are wasteful. Multiple failed Archive openings can exhaust attempts without deployment.
  • No double island click. Once the corporate map is proven, do not click the same island coordinate again.
  • Do not weaken route gates to save a failed timing/proof gate. Reset and diversify the opening instead.

Preview Rule

  • Preview beats shopping: once a red region opens a current mission preview, click the highlighted 8x8 preview board instead of comparing missions.
  • Preview target: the small highlighted 8x8 board is stable per preview on the current layout; hover may prove it in data mode, but hot runs click the named target.
  • Grid coordinate: on the current macOS layout, the preview-grid commit point is window (800,470). Old point (680,360) selects map territory behind the preview and is unsafe.
  • No pre-dismiss: do not clear the top advisor textbox before the grid click unless a new trace proves it is required for that screen.
  • One cheap retry maximum only when the first preview is unknown, no Start click happened, and the timer is still under the opening ladder.
  • Mission names are not speed vetoes. Dam, Satellite, Train, Tides, Volatile, and similar previews are playable when current preview proof and post-Start deployment proof exist.
  • Preview uncertainty is the blocker. Stale bridge data, wrong-region proof, impossible counts, or no deployment handoff means stop or reset.

Reset Matrix

Condition Decision Reason
Unknown or stale preview before Start, timer under 0:35, distinct cheap candidate exists. One revalidated same-island retry. Cheap second proof can beat a full setup reset only while the opening is still alive.
Unknown/stale preview at 0:35+, retry budget exhausted, or no distinct candidate. Abandon to setup. More preview work is now slower than rerolling the slate.
Route mismatch or uncertain identity appears after Start, deployment, or combat signal. Ignore the stale identity in speed mode and continue from verified deployment/combat proof. After mission entry, the mismatch is usually stale route metadata; rerolling wastes more timer than solving the board.
Ordinary mission over 3:00, first island over 15:00, or second island not started by 16:45. Reset unless terminal success is already proven. The 30:00 achievement margin is no longer healthy enough for boring reliable play.

Route Policy

  • Click a valid red mission quickly; do not spend live timer ranking a better red mission after a preview is proven.
  • Current preview proof plus deployment proof is enough to play; the solver owns combat difficulty.
  • Unknown OCR, stale preview, wrong-region clicks, setup mismatch, terminal panels, and safety gates remain hard stops.
  • The calibrated preview-board click is the primary commit path; visible Start text is for fallback evidence.
  • Wrong-region clicks and stale previews are run-health failures, not quirks to play through.

Combat Policy

  • Use the hot path from clean combat: lightning_loop --time-limit 2 --speed-loss-policy --destroy-time-pods --quiet.
  • Solve while paused through Lightning wrappers; ordinary bridge commands need a fresh heartbeat after unpause.
  • Execute bridge-first with no pause between mech actions by default, verify predicted diffs, and re-solve after mismatch.
  • Accept boring grid-safe plans over clever low-margin plans.
  • Reset Turn is for paused, proven tactical misses only. If the confirmation prompt is not visible, recover by fresh read/solve or restart.

Reward + Shop

  • Clear deterministic panels only after terminal OCR safety checks.
  • Ignore optional objectives, pods, and reputation efficiency unless free.
  • Read Grid Power, buy Grid Power until max or reputation is exhausted, then exit shop and leave island immediately.
  • No browsing, no reading item text live, no speculative purchases.

End Turn

  • Use emitted window-local click batches or calibrated controls, never legacy global coordinates.
  • If the confirmation modal appears, click the named confirmation control explicitly.
  • Retry End Turn only when bridge samples still prove the same player turn with no active mechs.
  • After enemy activity, wait for phase == combat_player, active_mechs > 0, and a stable roster before solving.

Speed Losses

  • Allowed only for active Lightning War attempts and only when the timeline survives.
  • Can trade pods, optional objectives, reputation, nonlethal buildings, or nonlethal grid when grid remains above zero.
  • When the selected top plan is already accepted by this policy, skip emergency clean-plan widening and record the skipped widening reason in the solve output.
  • Never trade mech HP, mech status, mech death, bombs, pylons, final-cave survival, unknown loss kinds, stale boards, research gates, or desyncs.
  • Destroy or ignore time pods by default; pod rewards add UI time and do not help Lightning War.

Readiness Tests

  1. Wrong-window dry run refuses to click when Codex or another app is frontmost.
  2. Pause proof captures two stable paused frames with game bounds and timer OCR.
  3. Microburst resumes and re-pauses within expected visible timer delta.
  4. Route preview refuses Start without current region/preview proof and a calibrated commit target.
  5. Deployment cannot be accepted as a generic pause state.
  6. Stale screenshot paths and mismatched bounds are rejected.
  7. Overlay simulation emits no route/start/confirm clicks.
  8. First-60-seconds trial repeatedly reaches confirmed deployment by <=0:20.

Telemetry

  • Every segment emits compact JSON: state, visible timer, focus proof, screenshot path, click target, and timer delta.
  • Summary rollups include segment results, speed-phase timing, pace gates, and attempt restarts.
  • Rolling screenshot samples can be pruned, but pinned proof frames cannot.
  • Non-game captures are stored separately as failure evidence.
  • Timing notes track island click, preview open, Start, Confirm, first player turn, each End Turn, tail clear, shop, and leave island.
  • The run declares success only after achievement pop, offline log proof, colored middle Blitzkrieg achievement tile, or synced unlock proof is captured.

After Every Attempt Record Next Code Decision

Every failed attempt also needs a machine-readable ledger row marked patched, documented_no_code_change, route_cache_updated, timing_updated, or discarded_with_reason.

Data Cadence

  • Race mode: screenshot every 5 seconds plus every transaction boundary when intentionally racing.
  • UI experiment mode: screenshot at 0.5 seconds / 2 Hz inside the rolling buffer; use 2 Hz whenever the next click timing is still being learned.
  • Keep the buffer small: default sample cap is 360 PNGs; archive only named proof frames outside the cap.
  • Mark pruned samples: frame rows whose images are removed cannot remain proof links.
  • Every failure needs one sentence: what cost the timer, what proof showed it, and what code should change.

Success Proof

  • Stop after the second Corporate Island boundary; do not choose a third island before checking for the unlock.
  • Capture completion-screen proof, visible final timer under 30:00, current squad, difficulty/content proof, and completed-island count.
  • If the pop is not visible, check the offline Into the Breach log and pause-menu achievement tile while the game is safe.
  • Require Lightning War in the on-screen pop, offline ITB log, colored middle pause-menu achievement tile, or synced unlocked_list.
  • Declare completion only when there is no KIA, failed-objective, timeline-loss, or setup-mismatch evidence.

Pinned Proof Frames

  • Pin setup verification, first island/map, route preview before Start, post-Start deployment proof, Confirm/first player turn, every End Turn boundary, mission tail, shop/leave, blocker, and success frames.
  • Copy any temp screenshot proof into the run telemetry screenshot folder before citing it.
  • Pinned frames are exempt from rolling retention and must keep stable links in notes and summaries.

Failure-To-Code Loop

  1. Preserve pinned evidence and classify the owner: capture, focus, route OCR, route click, deployment, bridge, combat, reward/shop, timer gate, or success proof.
  2. Write one hypothesis and one code target.
  3. Add or update a focused regression when possible.
  4. Run focused proof before the next live race, unless the ledger row is explicitly documented_no_code_change.
