# Agent Safety Gates

Use this file whenever a turn is blocked, dirty, unknown, desynced, or under investigation. These rules intentionally stop progress until the model gap or process risk is understood.

See also `docs/self_healing_loop_design.md`, `docs/diagnosis_loop_design.md`, and `docs/solver_goal_principles.md` for deeper design context.

## Gate Summary

- Never solve past unknown research. Snapshot, run `research_next`, capture/submit, attach community notes when requested, and repeat until no work remains.
- Never click a held End Turn batch after `INVESTIGATE`, `INVESTIGATE_POST_ENEMY`, `THREAT_AUDIT_BLOCKED`, or a persistent `post_enemy_block` until the miss is understood or explicitly resolved.
- After a resume, compaction, interruption, or any handoff ambiguity, run a fresh standalone `read` before dispatching a remembered End Turn click. If the live board is already a new player turn, discard the stale click plan.
- Dirty plans require the exact one-use `dirty_consent_id`, but the user has granted standing consent to execute reviewed dirty lines when they are not caused by a desync/investigation gate. Timeline collapse is not covered by standing consent; the documented final-cave Hail Mary needs an explicit live user authorization for that exact resist line.
- Drain diagnosis entries between turns, one at a time unless the user asks to clear the queue.

## Detailed Safety Rules

2. Never execute mech N+1 before the verify for mech N returns PASS, unless you consciously decide to log+continue past a desync.

19. **Always follow the solver — never override.** The solver encodes push directions, bump damage, projectile re-aiming, terrain interactions, env_danger ordering — your manual analysis will be wrong or incomplete. Execute what `game_loop.py solve` outputs. When `verify_action` reports a desync, the failure_db record is the useful signal (it feeds the auto-tuner). Overriding the solver suppresses that data. Only exception: solver returns empty (timeout) — play manually via screenshot reasoning. Rationale: `feedback_trust_solver.md`.

20. **Research gate — never solve past an unknown.** When `game_loop.py read` flags `requires_research: true`, or when `solve` / `auto_turn` return `error: RESEARCH_REQUIRED`, the self-healing loop has detected a pawn type, terrain, weapon, or screen we've never catalogued. Solving past it produces confidently-wrong plays. Do this before resuming combat: (a) `game_loop.py snapshot research_gate_<turn>` for rollback, (b) `game_loop.py research_next` to get the MCP capture plan, (c) inspect the batch before dispatch, then dispatch `plan.batch` via `mcp__computer-use__computer_batch`, zoom each `plan.crops[i].region`, run Vision with each bundled `prompts[crop_name]` to produce JSON per crop, (d) `game_loop.py research_submit <research_id> '<json>'`. Unit research during `combat_player` must be hover-only and click-free. If a research batch contains `left_click` while any active player actor remains, that is a hard stop: do not dispatch it, and obtain a corrected hover-only plan instead. After a partial action or desync, visible modal, dialogue, reward, or terminal UI outranks a stale bridge report of `combat_player`; do not use a research click to dismiss that UI or advance the mission. If the submit result includes `community_queries.queries`, (e) `WebFetch` each URL (Steam discussions + r/IntoTheBreach), summarize each into `{url, excerpt, confidence}` and (f) `game_loop.py research_attach_community <research_id> '<notes_json>'` to persist the corroborating community notes. After a display-name submit (e.g. "Centipede") for a tiered bridge type (e.g. `Centipede1`), ensure the exact bridge type(s) are present in `data/known_types.json`; otherwise `research_next` can say `NO_WORK` while `auto_turn` still gates on the tiered type. Repeat until `research_next` returns `status: NO_WORK`. Only then `auto_turn`. Rationale: `docs/self_healing_loop_design.md` §Missing wire.

21. **Codex Computer Use coordinate caveat.** `research_next`, legacy `grid_to_mcp` plans, and `tile_hover.py` emit MCP/global coordinates for the old batch tool. The Codex Desktop `mcp__computer_use__.click` tool expects screenshot/window-local coordinates, so don't paste those MCP coords directly into it — use the emitted `codex_computer_use_batch` when present, the visible tile center from the screenshot, or a helper that explicitly says its coords are Codex window-local. This came from a 2026-05-04 research miss where an MCP coord intended for `Snowtank1` selected the wrong visible unit, and recurred on 2026-05-09 when a `tile_hover.py G1` coordinate selected a building instead of `Starfish2`.

23. **Grid-drop investigation gate — fire now, not later.** When `auto_turn` returns `status: "INVESTIGATE"` or `INVESTIGATE_POST_ENEMY`, the solver's prediction lost grid power in a way the sim didn't model, or the harness knowingly advanced past a live threat. This is exactly how Prime_ShieldBash and Ramming Engines' empty-charge self-damage bugs surfaced — we missed the signal for multiple runs. Never click `pending_end_turn_batch` before resolving. For each entry in `investigations`: (a) read the `snapshot_path` (has `predicted.json`, `actual_board.json`, `context.json` with `grid_power_diff` + `building_hp_diffs` + `failure_db_id`), (b) spawn an Explore agent with those three files and the prompt "trace this unpredicted grid loss to a weapon-def, simulator, or push-chain bug — propose a fix with file+line and confidence level", (c) if the agent returns a concrete fix with high confidence, apply it, rebuild Rust if `rust_solver/src/*.rs` changed, run `bash scripts/regression.sh` to verify nothing else regressed, (d) if fix applied, `replay <run_id> <turn>` on the new solver to confirm it now predicts the actual grid power. Only then dispatch `pending_end_turn_batch` via `computer_batch`. If the agent cannot propose a fix (low confidence / needs more data), log that in the decision log, dispatch the batch, and continue — not every mystery is fixable mid-turn. If a `THREAT_AUDIT_BLOCKED` plan is advanced anyway because Reset Turn is visibly unavailable and the threatened tile is not mission-critical, assume the next `auto_turn` will raise `INVESTIGATE_POST_ENEMY`; immediately inspect the post-enemy record, log the exact accepted threat, clear it only with `resolve_post_enemy_block --reason`, and do not treat "Perfect Island can tolerate grid loss" as a reason to bypass this audit silently. Regression anchor: Perfect Strategy run `20260517_170505_240`, `Mission_BeltRandom` turn 3, Hornet1 F5 destroyed non-objective G5 after Reset Turn only hovered.

24. **Diagnosis loop — "drain the queue", "run the loop", "diagnose this run" all mean the same protocol.** When `verify_action --diagnose` is on (or env `ITB_AUTO_DIAGNOSE=1`), desyncs accrue in `session.diagnosis_queue`. The harness — that's you — drains the queue **between turns**, NEVER inside `auto_turn` (45s enemy-animation budget; design doc §13 #17). Per-call protocol:

    a. **Drain one entry.** `python3 game_loop.py diagnose_next` pops one pending failure, runs Layer 2 (rejections → known_gaps → rules.yaml). Outputs `rule_match` / `needs_agent` / `insufficient_data` / `rejected` / `EMPTY`. On `EMPTY`, stop and tell the user — don't spin polling.
    b. **`needs_agent` → dispatch.** Read the `## Agent prompt` block from `recordings/<run_id>/diagnoses/<failure_id>.md`, dispatch via the available agent/delegation tool with an exploration-oriented subagent. The validator tolerates prose-wrapped + multi-`{...}` agent output (last block wins, braces inside string literals are skipped) but enforces `target_language=rust`, every `suspect_files[*].path` resolves, every cited line exists, `confidence ∈ {high,medium,low}`, `fix_snippet.before` and `.after` both non-empty. Pipe the JSON via `python3 game_loop.py diagnose_apply_agent <id> <json|path>`. On validator failure: read the error list, fix the JSON OR `reject_diagnosis` if the proposal is bad on its merits.
    c. **`rule_match` → leave alone.** Rule fixes are conceptual prose (e.g. "guard set_active(false) on weapon_id != WId::None"), not Edit-applicable. Surface the rule_id + suspect_files to the user; `apply_diagnosis` will refuse a rule_match anyway.
    d. **`agent_proposed` → dry-run, then apply under standing desync-fix authorization.** Do not blindly auto-fire `apply_diagnosis`: it mutates source files, atomically bumps SIMULATOR_VERSION in BOTH `rust_solver/src/lib.rs` and `src/solver/verify.py`, archives `failure_db.jsonl`, runs `bash scripts/regression.sh`, and reverts on failure. Always run `apply_diagnosis <id> --dry-run` first to inspect the plan. If the dry-run is clean, the proposal is concrete/high-confidence, and the dirty-workdir guard passes, apply it without asking again; the user's standing live-loop preference is "if it is a desync, investigate and fix it." Multi-location fixes (function signature changes that need both def + call sites edited) currently can't be expressed in one `fix_snippet` — design doc §15.1 known limitation; reject those with a reason citing it.
    e. **`apply_diagnosis` refused on dirty workdir → surface, don't auto-resolve.** Refusal is by design (don't clobber user work). Print the dirty paths to the user; do NOT `git stash` or `git commit` on their behalf to "make it apply". Per AGENTS.md "Executing actions with care".
    f. **`apply_diagnosis` succeeded → verify, commit, and push.** After a diagnosis fix applies cleanly, rebuild/install Rust if touched, run the focused regression/replay that proves the desync is fixed, and run the broader regression harness when live timing allows. Then stage only the fix, simulator-version/failure-db snapshot, regression, and AGENTS.md guard changes; commit with the failure_id and agent confidence in the message, and push the current branch. Do not include unrelated dirty paths or generated raw snapshots that fail `git diff --check`.
    g. **`reject_diagnosis <id> --reason "..."` is the feedback channel.** Use it whenever a proposal is wrong (incomplete fix, wrong direction, would not compile, regresses corpus). The (diff_signature × sim_version × proposed_fix_sig) gets recorded in `diagnoses/rejections.jsonl` so the same wrong proposal can't recur. `--force` on diagnose bypasses, for when a rejection turns out to have been premature.
    h. **One drain per call** unless the user says "drain all" / "clear the queue" — agent dispatches cost real tokens. Report queue depth and ask before continuing. Exception: if every pending entry resolves to `rule_match` or `insufficient_data` (no agent dispatch), drain them all in one pass — they're cheap.

    Full spec: `docs/diagnosis_loop_design.md`. Live-run hardening retrospective: §15.1.

26. **Session lock discipline:** never run `game_loop.py read`, `status`, `solve`, `deploy_recommended`, `auto_turn`, `diagnose_next`, `diagnose_queue`, `research_next`, `research_submit`, `research_resolve`, `achievements`, `recommend_squad`, `new_run`, or other session-touching game-loop commands in parallel. They share or influence live session state; run them one at a time, then parallelize only independent file reads or analysis commands. Before every `multi_tool_use.parallel` call, scan each tool command string; if any contains `game_loop.py`, split it out into its own terminal call first, even for `status`, `log`, `-h`, or a command you believe is read-only. Do not tuck even a harmless-looking `status` / `recommend_squad` / `diagnose_queue` call into `multi_tool_use.parallel` beside `rg` / `sed` / `git status`, beside helper scripts such as `island_select.py`, `tile_hover.py`, or file reads used during research targeting, or beside Computer Use `get_app_state` / screenshot reads; issue the game-loop command separately, then do UI inspection or file/helper reads after it exits. During research-gate cleanup, run `diagnose_queue` as its own terminal call before or after `rg` / `jq` checks, never in the same parallel batch. Do not run `research_peek`, `research_next`, `research_resolve`, or stale-known-type checks in `multi_tool_use.parallel` beside `rg` / `jq` / `sed` / file reads; run the game-loop command alone, then inspect files separately. Do not chain `research_peek` / `research_next` / other read-only-looking research commands after `jq`, `rg`, or `sed` in one shell line; run the game-loop command alone, then inspect files in a separate call. If a remembered command such as `research_queue_peek` is not listed by `game_loop.py -h`, do not probe it inside a parallel batch; check or run the intended real command alone. This also applies during documentation/commit cleanup: if you need `status`, run it alone first, then fan out file reads afterward.

28. **Do not chain session-mutating commands.** Even with `&&`, commands like `py_compile && game_loop.py read` make live logs harder to audit and can hide which step changed session state. Run formatting/compile checks and game-loop commands as separate terminal calls. This includes one-off research cleanup checks: run any `rg` / file validation first, then run `game_loop.py research_resolve` in its own command. Do not pipe `game_loop.py` output into `sed` / `head` / `jq` / similar filters either, even for `-h`; run the `game_loop.py` command alone, then inspect captured/source output separately.

30. **Safety block emergency protocol.** On a safety block, `cmd_solve` now automatically widens to `solve_top_k(..., 100)` and, if the session has temporary soft-disabled weapons, runs a second emergency top-k pass with `disabled_actions` removed from the payload. This does **not** mutate `session.disabled_actions`; it only lets the current candidate search recover a clean plan. If it still returns `SAFETY_BLOCKED`, inspect the dirty frontier and execute the best reviewed dirty line under standing consent with `--allow-dirty-plan` plus the exact dirty token (see rule 37). On the final bomb turn, a dirty plan is acceptable only when the predicted grid remains above 0, the Renfield Bomb survives, and no clean candidate is found. A final-cave pylon loss is allowed only as an emergency exception after a deep search finds no pylon-preserving candidate, predicted grid remains above 0, all mechs survive without HP/status/danger debt, no other objective loss is predicted, and the exact dirty token is used. If `grid_timeline_collapse` appears, use rule 126 instead; standing dirty consent is insufficient.

31. **Research queue cleanup.** If `research_next` reports stale pending known-type entries, use `python3 game_loop.py research_resolve <TYPE> --kind behavior_novelty --reason "<why>"` rather than hand-editing `sessions/active_session.json`. The command validates `<TYPE>` against `data/known_types.json` and records a `manual_resolved` result with attempts/diff metadata.

32. **Research queue peek still matters.** `research_queue_peek` is a JSON result field, not a command. For a standalone read-only check use `python3 game_loop.py research_peek`; to process work use `research_next` / `research_submit`. If `auto_turn` completes but its `research_queue_peek` contains a current-board unknown such as `BigBomb`, snapshot and resolve that queue item before advancing another turn, even if no hard `RESEARCH_REQUIRED` gate fired.

36. **Enemy attack intent must be save-gated when available.** `GetSelectedWeapon() > 0` can stay nonzero on non-attacking Vek after movement/re-aim animation, causing Rust's missing-target phantom attack guard to invent false building damage. When `saveData.lua` exposes `iQueuedSkill`, bridge code must derive `has_queued_attack` from `iQueuedSkill >= 0` and only fall back to `GetSelectedWeapon()` when the save field is missing. Regression anchor: Normal run `20260504_210332_088`, mission 7 turn 3, Firefly1 uid `33` falsely threatened B5 and produced a false safety block; fixed in bridge + simulator v47.

37. **Dirty-plan acceptance is exact, single-use, and standing-approved unless desynced.** `--allow-dirty-plan` by itself is not enough for a blocking plan. `solve` / `auto_turn` emits a `dirty_consent_id` tied to run, mission, turn, candidate rank, input fingerprint, loss profile, and exact action list. The user has granted standing consent to execute reviewed dirty lines, so after inspecting the frontier, rerun the chosen exact line with `--allow-dirty-plan --dirty-consent-id <id>` without asking again. If the CLI requires a broader dirty flag for that exact loss class, include only the required flag for the reviewed line. A live desync, `INVESTIGATE`, `INVESTIGATE_POST_ENEMY`, `THREAT_AUDIT_BLOCKED`, persistent `post_enemy_block`, unresolved research, or board uncertainty overrides standing dirty consent: stop, snapshot, investigate, and fix/resolve before continuing. Predicted mech loss is not dirty-consentable, even on a final mission turn or speed achievement route; preserve the pilot/mech or restart the attempt. Predicted timeline collapse is not dirty-consentable under standing consent; final-cave pylon loss is dirty-consentable only under the grid-positive emergency exception in rule 30. For a final-cave resist gamble with predicted timeline collapse, stop and ask for explicit authorization for that exact live line.

39. **Controllable mission allies count if they have weapons.** Friendly non-mech units such as `Archive_Tank` can be READY and player-controlled even when `mech=false` in bridge JSON. If `read` shows a READY friendly with a weapon and the solver omits it, first check whether that weapon has a Rust `WId` mapping. `Deploy_TankShot` / Stock Cannon is a 0-damage projectile with forward push; ignoring it can turn a salvageable tank mission into a false grid-zero safety block.

40. **Dirty frontier before dirty execution.** When a solve safety-blocks and reports `dirty_frontier`, inspect the tradeoff classes before spending the standing dirty consent token. The top raw-score dirty plan may not represent the best strategic compromise; compare options such as `grid_loss`, `mech_loss`, `building_loss`, and `objective_loss` explicitly, especially on Normal where a single dirty Turn 1 can cascade into timeline loss.

41. **Lookahead frontier is diagnostic until promoted.** `project_plan_scenarios` returns bounded plausible next-turn enemy-intent scenarios: base `heuristic_requeue` plus a few high-value building retargets. `lookahead_frontier` in solve output previews worst projected recovery across those scenarios, and `robust_frontier` ranks tradeoffs by current score + worst next-turn score, but neither selects live actions yet. Treat both as evidence for dirty-plan triage and future robust-beam tuning, not as permission to bypass safety.

45. **Manual plan scoring can be invalid-action dirty.** `score_plan` / `replay_solution` are diagnostic tools, not action validators. If a hand-written plan produces an `illegal_*` replay event such as `illegal_leap_landing:x:y:unit`, the plan is impossible in-game and must not be used as dirty-plan consent evidence. Regression anchor: Easy run `20260506_114649_974`, Disposal Vaults turn 4 false clean where Aerial Bombs was hand-scored onto live Spider2 at D2; fixed in simulator v57 by no-oping illegal Leap landings.

47. **Soft-disable masks must cover every WId.** New weapon IDs can exceed 127; if the disabled-action bitset only has one `u128`, a desynced high-ID weapon will not actually be suppressed on the next solve. Keep disabled masks wide enough for all `WEAPON_COUNT` entries and add/keep a regression whenever a newly catalogued weapon is above 127. Regression anchor: `Missiles_OneDmg` WId 135 stayed selectable after a turn-1 desync until simulator v59 widened the mask.

51. **Post-action desyncs are an automatic investigation gate.** Any `verify_action` / `auto_turn` desync after a live bridge action (`click_miss`, `death`, `push_dir`, status/tile mismatch, etc.) must be investigated before resuming gamer-mode or selecting the next mission, even if the visible outcome is Region Secured or otherwise successful. Do not dismiss it as harmless because the mission survived. Minimum protocol: inspect the saved `*_action_*_verify.json`, the newest `recordings/failure_db.jsonl` entry, and the corresponding `*_solve_input.json`; reproduce with `itb_solver.solve` / `solve_top_k` when target legality or search behavior is suspect; decide whether it is a known benign bucket, stale UI/bridge timing, or a concrete solver/simulator/control bug. If a concrete bug is found, fix it immediately, add/keep a regression, rebuild the Rust wheel when `rust_solver/src/*.rs` changed, run the focused replay/test that proves the desync is fixed, run the broader regression harness when live timing allows, update AGENTS.md with the new guard, then stage only the relevant changes, commit, and push before continuing. This commit/push step is mandatory for verified solver/simulator fixes unless the user explicitly pauses it; do not carry a fixed-but-unpushed solver into the next mission.

52. **Partial re-solves must preserve upgraded weapon overlays.** `cmd_solve` patches bridge mech loadouts from save-file weapons because Lua reports base SkillList IDs, but `_re_solve_partial` must do the same after a desync. If an upgraded weapon such as `Science_Repulse_A` predicts base behavior after a re-solve (for example Pulse does get Shield Self in-game but predicted `status.shield=false`), check that `_enrich_bridge_mech_weapons_from_save` runs in the partial re-solve path before Rust solve/replay. Regression anchor: Easy run `20260506_114649_974`, Corporate HQ turn 1 after RocketMech status desync, Pulse Repulse predicted no shield while actual shield appeared; fixed in simulator v63.

56. **Perfect Battle turns treat mech HP loss as blocking.** While explicitly chasing Perfect Battle, any `plan_safety` / dirty-frontier `mech_hp_loss` must stop before the End Turn click, even if the solver labels it `WARN` or the HP display looks over-max from pilot bonuses. Spawn-blocking damage, bump damage, fire, and any other mech HP drop all void the achievement. Do not rationalize the warning away; either find a no-mech/no-building-damage plan or explicitly abandon the Perfect Battle attempt before advancing. Regression anchor: Easy Rusting Hulks run `20260508_122657_124`, Mission_FreezeBots turn 1 predicted `mech_hp_loss` before End Turn and then actual Rocket/Pulse damage on enemy phase.

56a. **Lightning War speed routes do not buy mech HP or status.** While explicitly chasing Lightning War, speed policy may trade optional objectives, pods, reputation, buildings, or grid only while the timeline survives. Any predicted `mech_hp_loss` or new mech status debt must safety-block before execution, even if the loss clears mites, blocks a spawn, or saves seconds. Abandon/restart instead of clicking End Turn or running the warning plan. Regression anchor: Blitzkrieg Lightning War run `20260603_133757_212`, R.S.T. `Mission_Tides` turn 1, where a non-blocking `mech_hp_loss` warning executed and damaged WallMech while clearing mites.

58. **Retire completed achievement blockers immediately.** `achievement_targets` can outlive the unlock inside an active session. After `achievements --sync` marks a target complete (including Steam client-cache promoted fresh unlocks), chase-specific safety blockers for that target must stop applying so the same timeline can pivot to the next objective. Regression anchor: Easy Rusting Hulks run `20260508_134925_472`, Scrapyard turn 2 was safety-blocked by Perfect Battle mech-HP rules after `Ach_Rust_A_3` had already unlocked.

59. **AE Moth/Bouncer recoil is a real queued self-push.** `MothAtk1` / `MothAtk2` and `BouncerAtk1` / `BouncerAtk2` queue a backward self-push before their target hit, then push the target forward. A blocked recoil bumps both the attacker and the blocker, so a mech behind a Moth can lose HP even when the artillery target is elsewhere. If post-enemy HP is one lower than predicted around Moths/Bouncers, check self-push ordering before blaming spawn-blocking or HP display noise. Regression anchor: Easy Rusting Hulks run `20260508_134925_472`, Scrapyard turn 2 Moth at E4 fired at E7 while PulseMech stood behind it at E3; live Pulse took recoil bump plus spawn-block damage; fixed in simulator v68.

61. **Dirty-plan selection should minimize same-class collateral.** If every candidate is safety-blocked and the dirty frontier has the same irreversible loss class (for example all lose exactly 1 grid / 1 building HP), do not blindly preserve the highest raw-score plan when it also causes avoidable mech HP loss. Widen the dirty frontier enough to inspect lower-ranked candidates and choose the same blocking-loss profile with the least non-fatal mech HP collateral before asking for dirty-plan consent. Regression anchor: Easy Rusting Hulks run `20260508_134925_472`, Scrapyard turn 4 top dirty candidates nicked PulseMech, while rank 110 kept all mech HP intact with the same unavoidable grid/building loss.

74. **Aerial Bombs bridge transit damage is adaptive.** `pawn:FireWeapon()` may or may not apply Aerial Bombs / Bombing Run transit terrain damage before the bridge workaround runs. The Lua bridge must snapshot transit terrain HP/state before `FireWeapon` and synthesize damage only when the engine left that tile unchanged; otherwise a 2-HP mountain/objective can be double-damaged. A mountain-only `building_hp` diff is a terrain desync, not a grid-drop gate. Regression anchor: Easy Rusting Hulks run `20260508_134925_472`, Archive `Mission_Repair` turn 1: JetMech C3→C1 over C2 mountain became rubble after bridge double-applied `Brute_Jetmech` transit damage.

78. **Direct building damage drains grid per HP; bump exceptions stay narrow.** Direct weapon/explosion building damage reduces current Grid Power by each building HP lost, even when a 2-HP building survives at 1 HP. Non-unique multi-HP push/bump collision damage remains the narrow exception: it preserves grid on partial damage and drains only on destruction, while unique/inferred objective buildings drain per HP on bump. Beware the animation gap: a first bridge read can show `phase=combat_player` with `active_mechs=0` and stale grid before the delayed grid drop settles. Regression anchor: Easy Rusting Hulks run `20260508_134925_472`, Archive `Mission_Dam` / Central Museums turn 2: Blast Psion aura damaged a 2-HP D3 building 2→1, and the grid dropped 5→4 only after the player phase fully activated; corrected in simulator v85 plus post-enemy recording guard.

80. **Blobber Leader is not an Alpha Blobber alias.** `BlobberBoss` uses `BlobberAtkB` to spawn a 2-HP minor `BlobB`; that Blob Leader uses `BlobAtkB`, taking 1 self damage while dealing 2 damage to the four cardinal adjacent tiles. The split inner/outer damage matters: aliasing to `BlobberAtk2` / `BlobAtk2` predicts the wrong spawned pawn and often kills the Blob Leader too early. Regression anchor: Easy Rusting Hulks run `20260508_134925_472`, Archive Corporate HQ turn 1 research gate found `BlobberBoss` at F2 and `BlobB` at F6; fixed in simulator v87.

81. **Move-status effects can settle after bridge MOVE.** A bridge `MOVE` can return before tile fire status finishes applying to the moved pawn; if a per-sub-action verify sees only predicted-true / actual-false unit status diffs, refresh and re-read briefly before logging a simulator desync. Flying units still catch ordinary tile fire once the state settles. Regression anchor: Easy Rusting Hulks run `20260508_134925_472`, Mission_Final_Cave turn 4: JetMech moved H3→F4 onto burning ground, immediate verify saw `fire=false`, but a settled bridge read showed `fire=true`; handled in simulator v89 plus auto-turn settle retry.

85. **Diagnostic score_plan must include spawn blocking.** `score_plan` is only an audit helper, but its post-enemy summary must still mirror replay/project_plan ordering: mech actions → enemy attacks/environment → spawn blocking. If a hand-scored repair or shield plan looks clean while the solver says a mech dies on an emerging Vek tile, first suspect missing spawn-blocking in the diagnostic path rather than a solve miss. Regression anchor: Easy Zenith Guard run `20260510_000847_773`, Mission_Volatile / Erosion Flats turn 2: webbed 1-HP Defense Mech at F3 repaired to 2 HP, then still took Scorpion damage plus spawn-blocking damage; fixed in simulator v91.

86. **Shield Projector is a building-defense action, not just unit support.** `Science_Shield` is `LineArtillery`: it can target live building tiles, shields the target plus the next tile in the shot direction, and a shielded building consumes the next damage instance before losing HP/grid. Do not filter shield artillery targets with the normal "never target buildings" artillery guard, and carry building tile shields in Rust tile flags/bridge extraction. Regression anchor: Easy Zenith Guard run `20260510_000847_773`, Corporate HQ `Mission_BurnbugBoss` dirty-chain deep dive: the solver could not consider shielding C3/F3-style building threats, forcing repeated dirty plans; fixed in simulator v92.

90. **Safety widening must be wide enough before dirty consent.** When `auto_turn` safety-blocks, the emergency top-k pass must search deep enough to catch lower-ranked clean plans before asking for `--allow-dirty-plan`. Rank-515 clean Rift Walkers plan (PunchMech F4→E5 punches D5, TankMech fires F6, ArtiMech D7→C6 fires H6) was missed by a 500-candidate widening window and incorrectly looked like forced G5 grid loss. Keep `_SAFETY_WIDENING_TOP_K` at least 1000 unless a larger diagnostic replaces it. Regression anchor: Easy Rift Walkers run `20260510_213059_819`, Mission_Train turn 2.

94. **New Rust weapon IDs must refresh `known_types`.** After adding a `WId` variant or bridge-observed upgraded weapon ID, update `data/known_types.json` (or regenerate it and inspect the diff) before resuming live combat. Otherwise the research gate can block on a weapon the simulator already supports while `research_next` has no visual work to perform. Regression anchor: Easy Rift Walkers run `20260510_213059_819`, R.S.T.
