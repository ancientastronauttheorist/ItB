- The rolling sample was contaminated before combat by deployment tooling: `deploy_recommended` partially placed a mech and then timed out, leaving a live deployment screen. Manual deployment then cost roughly 60 seconds and confused the next conductor call because the bridge reported `combat_enemy` plus remaining deployment zones.
- `lightning_attempt` should treat `phase=unknown/combat_enemy`, `in_active_mission=true`, `deployment_zone_count>0`, and `mech_count<3` as an active deployment state, not stale preview/map data. It should finish the remaining deployment instead of waiting 30 seconds or asking for UI inspection.
- Acid Storm is a poor average-speed sample when the clock matters: it has live rain/environment animation, ACID Storm objectives, and early dirty lines. Turn 2 required accepting 1 grid/building HP loss; after executing the accepted line, threat audit correctly held End Turn because the remaining Firefly threat was exactly that consented grid hit.
- Tooling fix applied after this sample: `deploy_recommended` now skips already placed mechs, continues after a deploy ACK timeout only when a fresh read shows the UID placed, and `lightning_attempt` treats `turn=0` + deployment zones + `mech_count < 3` as missing-deployment recovery. A validated ordinary grid/building dirty consent also satisfies the final threat audit when the remaining threat count is covered by the reviewed loss profile.
- Additional deployment fix: on `Mission_Barrels`, the bridge logged mission start but never polled `DEPLOY`, which means deployment can be live UI before Lua command polling wakes up. `deploy_recommended` now uses a short bridge deploy timeout, clears stale paused `/tmp/itb_cmd.txt` commands before UI bursts, and falls back to calibrated deployment-slot + ranked tile clicks when the bridge is not ticking.
- Follow-on stale-state guard: after the timeout patch, the visible screen was actually the Detritus island map while the bridge still looked deployment-like. The UI classifier now detects visible island maps and `lightning_attempt` refuses to deploy from stale `phase=unknown` deployment zones when the screen is clearly map/routing.
- `Mission_Barrels` generated a Time Pod and stopped turn 3 on `pod_unrecovered_final`. For Lightning War this is process noise, not achievement value, so pod-only safety losses are now non-blocking for active Lightning War targets. Grid/building/objective/mech losses remain blocked.
- Detritus `The Heap` / `Mission_Barrels` route mechanics: clicking the region opens a CEO dialogue plus preview; dismissing the dialogue returns to the plain map. The reliable commit observed here was region click, wait for the dialogue/preview, then click the same region again without dismissing. This landed in deployment.
- `Mission_Barrels` timing was too slow for the 3-minute target: deployment appeared around `1:10:11`, and the paused Region Secured screen was `1:13:40`, roughly 3m29s before post-mission Continue/pod/map tail. It is useful for tooling, but avoid vats/barrels missions in a real Lightning War attempt when faster Train/Tidal/Cataclysm/Sandstorm options exist.

## 2-Minute Tooling Build Notes

- Built `lightning_segment` as the higher-level start-to-start conductor. It loops the lower-level attempt command, runs preflight once, keeps dirty consent pending until combat begins, clears safe panel chains, and stops on the next route/map decision while trying to park in pause.
- Added a Lightning War speed-loss policy for the real achievement chase: pod-only losses are expendable, and the segment path may also accept predicted nonlethal grid/building damage or optional objective/protected-unit losses when grid remains above zero and no mech loses HP/status or dies.
- Connected that same speed policy to the final threat audit, so a predicted accepted building/objective hit can cover matching `still_threatened` audit entries instead of forcing a slow exact dirty-token review.
- Kept the guardrails tight: timeline collapse, mech loss/status, pylon/bomb/final-cave losses, unknown loss kinds, research/desync/stale-board gates, and uncovered live threats still stop the loop.
